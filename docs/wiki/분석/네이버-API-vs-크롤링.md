# 네이버 API vs 크롤링 — 맛집 데이터 소스 비교

> "네이버 API로 이 정보를 받을 수 있을까?"에 대한 답: **부분적으로만 가능**. 현재 크롤링으로 얻는 데이터의 **약 20~30%**만 공식 API가 제공한다.

## 결론 먼저

| 선택 | 추천도 | 이유 |
|------|-------|------|
| **크롤링만** (현재 방식) | ⭐⭐⭐ | 필요한 모든 데이터 확보 가능, ToS 리스크 있음 |
| **API만** | ⭐ | 기본 정보만 가능, 리뷰/메뉴/평점 불가능 — AI 추천에 부적합 |
| **API + 크롤링 하이브리드** | ⭐⭐ | 기본 정보는 API, 상세는 크롤링 — 복잡도 증가 |

현재 시스템은 AI 상황 추천이 핵심이므로 **리뷰 감성, 메뉴 키워드, 블로그 요약** 이 반드시 필요 → 크롤링 유지가 현실적.

---

## 네이버가 제공하는 API 종류

### 1. 🔍 네이버 검색 API — `search/local.json`
**가장 관련 있는 API**. 지역 검색 결과를 JSON으로 반환.

- **엔드포인트**: `https://openapi.naver.com/v1/search/local.json?query=창원+맛집`
- **요금**: 무료 (일 25,000회)
- **인증**: `X-Naver-Client-Id` / `X-Naver-Client-Secret` 헤더
- **반환 항목**:
  ```json
  {
    "items": [{
      "title": "○○한식당",
      "link": "https://...",
      "category": "한식>한정식",
      "description": "",
      "telephone": "",
      "address": "경남 창원시 ...",
      "roadAddress": "경남 창원시 상남로 ...",
      "mapx": "128...",
      "mapy": "35..."
    }]
  }
  ```
- **제약**:
  - 검색당 **최대 5개**, 페이징 불가 (사실상 Top 5)
  - **리뷰/평점/사진/메뉴/영업시간 모두 없음**
  - category는 "한식>한정식" 수준 (세부 태그 불가)

### 2. 🗺️ 네이버 지도 API
- **지도 표시, 경로 안내, 지오코딩 전용**
- 맛집 상세 데이터 없음

### 3. 📝 네이버 블로그 검색 API
- `search/blog.json` — 블로그 글 제목/요약 반환
- 리뷰를 간접적으로 수집 가능하지만, 어느 가게의 리뷰인지 매칭 로직 별도 필요

### 4. ❌ Naver Place API (공식 상세 API)
- **일반 개발자에게 미공개**. 네이버가 직접 운영하는 공공/파트너 API는 있으나 공개 신청 불가능.
- 일본 HotPepper가 제공하는 식당 상세 데이터 레벨의 공식 API가 **한국에는 없음**.

## 데이터 항목별 비교

### 현재 크롤링으로 수집하는 항목

| 항목 | API 제공? | 비고 |
|------|---------|------|
| 이름 | ✅ local API | |
| 주소 (지번+도로명) | ✅ local API | |
| 좌표 (lat/lng) | ✅ local API | mapx/mapy |
| 카테고리 | 🔶 일부 | "한식>한정식" 수준만 |
| 전화번호 | 🔶 일부 | 빈 값 많음 |
| 대표 이미지 | ❌ | 없음 |
| **평점** | ❌ | API 없음 |
| **방문자 리뷰 수** | ❌ | API 없음 |
| **블로그 리뷰 수** | ❌ | API 없음 |
| **키워드 리뷰** ("음식이 맛있어요" 229명) | ❌ | 크롤링만 가능 |
| **메뉴 키워드** (오징어 38건) | ❌ | 크롤링만 가능 |
| **블로그 리뷰 본문 요약** | 🔶 | 블로그 API로 간접 |
| **영업시간 / 휴무일** | ❌ | 없음 |
| **편의시설** (룸, 주차, 키즈존) | ❌ | 없음 |
| **메뉴 가격** | ❌ | 없음 |
| **새로오픈 여부** | ❌ | 없음 |

**결론**: API는 **"이름/주소/좌표/카테고리"** 수준만 제공. AI 추천에 필요한 **신호 데이터의 70% 이상이 없음**.

## 하이브리드 전략 (검토 가능)

### 아이디어
1. **API로 기본 정보 수집** → 빠르고 안정적으로 10,000개 업체 확보
2. **크롤링으로 상세 보강** → 리뷰/메뉴/키워드/블로그 등

### 장점
- 검색 API는 ToS 이슈 없음 (공식 API)
- 기본 정보 확보 속도 빠름
- 크롤링 부하 감소

### 단점
- 구현 복잡도 증가 (두 소스 데이터 머지 로직 필요)
- 일 25,000회 제한으로 대규모 일시 수집 어려움
- 페이지당 5개 제한 → 다양한 키워드로 반복 호출해야

### 구현 시 코드 예시
```ts
// Naver Search API 호출
const res = await fetch(
  `https://openapi.naver.com/v1/search/local.json?query=${encodeURIComponent(query)}&display=5`,
  {
    headers: {
      'X-Naver-Client-Id': process.env.NAVER_CLIENT_ID!,
      'X-Naver-Client-Secret': process.env.NAVER_CLIENT_SECRET!,
    },
  },
);
const { items } = await res.json();
```

## 법적/윤리적 고려사항

### 크롤링 관련 쟁점
- **robots.txt 준수**: `https://pcmap.place.naver.com/robots.txt` 확인 필요
- **저작권**: 리뷰/블로그 본문 원문 그대로 저장 시 문제 가능 → **요약/변형** 원칙
- **과도한 트래픽**: 3초 이상 간격 + User-Agent 명시 + 병렬 제한
- **상업적 이용**: 약관상 명시적 금지 조항이 없으나 분쟁 가능성 있음

### 권장 사항
1. 리뷰/블로그는 **AI 요약**으로 가공 (원문 직접 노출 최소화)
2. 원문 링크 제공 (크레딧)
3. 수집 주기 여유 (일 1회)
4. 크롤링 실패 시 재시도 자제
5. 사용자에게 **데이터 출처 고지** ("네이버 플레이스 기반")

## 해외 비교

| 국가 | 공식 API |
|------|---------|
| 🇯🇵 일본 | HotPepper API (예약, 쿠폰, 시나리오 검색 포함) — 리크루트 제공 |
| 🇺🇸 미국 | Yelp Fusion API, Google Places API (리뷰·평점 풍부) |
| 🇰🇷 한국 | 네이버 Local Search API (기본 정보만) / 카카오 Local API (유사) |

→ **한국은 상세 맛집 API가 사실상 부재**. 크롤링 의존도가 높은 구조.

## 최종 권장

### 단기 (V2 MVP)
**크롤링 유지** + 과도한 리뷰 원문 저장 자제 + AI 요약 활용

### 중장기
- 네이버/카카오와 **B2B 파트너십** 타진 (공식 데이터 접근)
- **언니픽 V1 사장님 직접 입력 데이터** 확대 (1차 데이터)
- **공공 데이터** 병용 (모범음식점, 음식점 등록 현황)

### 확실히 피해야 할 것
- 대량/병렬 크롤링 (IP 차단 위험)
- 원문 리뷰 그대로 노출 (저작권)
- 로그인 세션/쿠키 탈취형 수집 (불법)
