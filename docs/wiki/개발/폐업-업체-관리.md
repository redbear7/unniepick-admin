# 폐업 업체 감지 및 관리 전략

> 맛집 데이터의 신뢰도 핵심은 "지금 영업 중인가"입니다. 폐업 업체가 추천에 섞이면 앱 신뢰도가 급락합니다.

## 핵심 원칙

1. **삭제 ❌, 상태 변경 ✅** — 히스토리 유지 (나중에 재개업 가능, 통계 데이터)
2. **즉시 반영 ❌, 단계적 의심 ✅** — 1회 이상 실패 시 바로 폐업 처리 말고 "의심" 단계 거치기
3. **자동 감지 + 수동 확인 병행** — 크롤러 판단 + 사장님 직접 보고 + 사용자 제보

---

## 감지 방법

### 방법 A: 네이버 페이지 재접근 (1순위, 자동)
주간 배치로 **모든 업체의 상세 페이지**를 다시 조회:

| 응답 | 해석 |
|------|------|
| 정상 로드 + 데이터 있음 | **영업중** |
| 404 / "존재하지 않는 업체입니다" | **폐업 의심 (강)** |
| 리다이렉트 to 다른 페이지 | **이전/통폐합 의심** |
| "영업종료" 텍스트 본문에 명시 | **폐업 확정** |
| 접근 실패 (네트워크/차단) | **불명** (재시도 대기) |

### 방법 B: 리뷰 활동 모니터링 (2순위, 자동)
- 최근 방문자 리뷰 / 블로그 리뷰 날짜 추적
- **90일 이상 신규 리뷰 0건** + 재접근 실패 → 폐업 의심

### 방법 C: 네이버 검색 결과 누락 (3순위, 자동)
- 원래 수집된 쿼리로 재검색 시 **누락된 업체** → 의심 플래그

### 방법 D: 사장님 직접 보고 (1차 데이터)
- V1 매장의 경우 사장님 앱에서 "영업 종료" 토글
- 사장님 보고는 **즉시 적용**

### 방법 E: 사용자 제보
- 사용자 앱에서 "폐업한 것 같아요" 신고 버튼
- 3건 이상 누적 → 의심 상향

### 방법 F: 공공데이터 교차검증 (4순위, 주기적)
- [공공데이터포털 - 전국 일반음식점 표준데이터](https://www.data.go.kr/data/15096283/standard.do)
- 월 1회 CSV 대조 → 폐업 여부 확인
- 정확도 높지만 갱신 주기 느림

## 상태 체계 (proposed)

```sql
-- restaurants 테이블에 컬럼 추가
ALTER TABLE restaurants ADD COLUMN IF NOT EXISTS operating_status TEXT DEFAULT 'active';
-- values: active | suspected | inactive | relocated | unknown

ALTER TABLE restaurants ADD COLUMN IF NOT EXISTS last_verified_at TIMESTAMPTZ;
ALTER TABLE restaurants ADD COLUMN IF NOT EXISTS closed_at TIMESTAMPTZ;
ALTER TABLE restaurants ADD COLUMN IF NOT EXISTS closure_source TEXT;
-- values: naver_404 | naver_closed_text | owner | user_report | public_data
ALTER TABLE restaurants ADD COLUMN IF NOT EXISTS closure_confidence SMALLINT DEFAULT 0;
-- 0-100 점수
```

### 상태 전이

```
active (영업중)
  │
  ├─ 1회 접근 실패 ─→ suspected (의심) [confidence: 30]
  │                       │
  │                       ├─ 2회 연속 실패 ─→ suspected [60]
  │                       │                       │
  │                       │                       ├─ 3회째 실패 ─→ inactive [90]
  │                       │                       │
  │                       │                       └─ 정상 복귀 ─→ active [reset]
  │                       │
  │                       └─ 정상 복귀 ─→ active
  │
  ├─ "영업종료" 텍스트 감지 ─→ inactive (폐업) [100]
  │
  ├─ 사장님 보고 ─→ inactive [100]
  │
  ├─ 주소 변경 감지 ─→ relocated (이전) [90]
  │
  └─ 네트워크 에러만 ─→ unknown (보류, 상태 유지)
```

## 크롤러 추가 모드: 주간 검증

```typescript
// 신규 npm script: npm run verify
// scripts/crawl-restaurants/src/verify.ts

// 흐름
for (const r of allRestaurants) {
  const result = await checkNaverPlaceStatus(r.naver_place_id);

  switch (result) {
    case 'ok':
      await updateStatus(r.id, { operating_status: 'active', last_verified_at: now });
      break;
    case '404':
      await incrementSuspicion(r.id, 'naver_404');
      break;
    case 'closed_text':
      await markInactive(r.id, 'naver_closed_text');
      await notifyTelegram(`🔴 폐업 감지: ${r.name}`);
      break;
    case 'network_error':
      // 상태 변경 없음
      break;
  }

  await sleep(3000 + Math.random() * 2000); // 과부하 방지
}
```

### 실행 주기
- 매주 일요일 새벽 03:00 (`launchd` 또는 cron)
- 1,000개 기준 약 1시간 소요 (3초 간격)

## 사용자 앱 (V2) 대응

### 영업중 필터 (default)
- 기본값: **`operating_status = 'active'` 만 노출**
- suspected는 기본 비노출, 관리자 토글로만 표시

### 폐업 플래그 UI
**inactive 업체를 검색 결과에 포함할 때**:
```
🔴 [폐업] ○○맛집
   2026-03-10 폐업 확인 | 네이버 감지
   (주변 대안: ...)
```

### 사용자 제보 버튼
- 맛집 상세 화면 하단 "폐업 신고" 버튼
- 3건 이상 누적 → 자동으로 suspected로 전환 + 관리자 알림

## 주의 사항

### 오탐(False Positive) 방지
- 네트워크 오류 1회는 **즉시 폐업 처리하지 말 것**
- 네이버 페이지 일시 장애 가능성 → 3회 연속 실패 시에만 inactive
- suspected 상태에서 **정상 응답 1회**면 바로 active 복귀

### 미탐(False Negative) 방지
- 크롤러 의존도 낮추기 — **사장님/사용자 제보 창구** 확보
- 정기적으로 "최근 리뷰 0건 + 6개월 이상" 업체 수동 검토

### 법적 고려
- "폐업" 표시는 **신중하게** — 실제 영업 중인데 폐업 표시 시 명예훼손 소지
- 3단계 확인 (자동 감지 + 시간 경과 + 관리자 승인) 후 공개 표시
